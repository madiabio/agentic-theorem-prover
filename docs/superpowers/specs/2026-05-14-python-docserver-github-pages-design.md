# Python Docserver and GitHub Pages Design

## Goal

Add Rustdoc-like Python API documentation for this repository, with a local docserver for development and an automated GitHub Pages deployment for published static docs.

## Recommended Approach

Use `pdoc` for the first version. It provides a low-configuration API reference generated from Python modules, docstrings, type hints, classes, and functions. This matches the desired Rustdoc-style experience more closely than a larger Sphinx setup.

The published site should include a small landing page plus generated API docs. The landing page gives visitors a stable entry point and can link to the generated API reference, `README.md`, and later report or experiment documentation.

## Dependency Management

Use `uv` instead of `pip` for local and CI workflows.

The repo should gain a `pyproject.toml` that declares runtime dependencies currently listed in `requirements.txt` and a docs dependency group containing `pdoc`. A generated `uv.lock` should be committed for reproducible local and CI installs.

Local usage should prefer:

```powershell
uv sync --group docs
uv run python scripts/build_docs.py
uv run python scripts/serve_docs.py
```

The first command installs project and docs dependencies. The second builds static HTML into `site/`. The third rebuilds the docs and serves them locally.

## GitHub Pages Workflow

Add `.github/workflows/docs.yml` with a GitHub Pages workflow that:

1. Checks out the repository.
2. Installs Python.
3. Installs `uv` through `astral-sh/setup-uv`.
4. Rejects package lines in `requirements.txt`.
5. Runs `uv lock --check`.
6. Runs `uv sync --group docs --locked`.
7. Builds static docs into `site/` with `uv run python scripts/build_docs.py`.
8. Uploads `site/` as a Pages artifact.
9. Deploys the artifact with `actions/deploy-pages`.

The generated `site/` directory should not be committed. It is CI output.

## Files and Responsibilities

- `pyproject.toml`: Project metadata, Python version, runtime dependencies, docs dependency group.
- `uv.lock`: Reproducible dependency lockfile generated by `uv`.
- `docs/index.md`: Short landing content used as the pdoc home page, if supported cleanly.
- `scripts/serve_docs.py`: Optional convenience wrapper around pdoc for local serving.
- `.github/workflows/docs.yml`: Build and deploy documentation to GitHub Pages.
- `.gitignore`: Ignore generated documentation output such as `site/`.
- `README.md`: Add a short "Documentation" section with local preview and GitHub Pages notes.

## Testing and Verification

Local verification should include:

```powershell
uv sync --group docs
uv run python scripts/build_docs.py
```

CI verification happens through the GitHub Pages workflow. The workflow should fail if docs cannot be imported or rendered.

## Error Handling

If pdoc cannot import a module because an optional runtime dependency is missing, the dependency should be declared in `pyproject.toml` rather than hidden in CI. If a module performs expensive work at import time, the implementation should avoid importing that module for docs or refactor the module so imports are side-effect-light.

## Open Decisions

The exact GitHub Pages URL depends on the repository owner/name and Pages settings on GitHub. The workflow can be added now, but Pages must be configured to use "GitHub Actions" as its source in the repository settings.
